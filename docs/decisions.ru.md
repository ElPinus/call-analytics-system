# Architecture Notes — Call Analytics System

> **Дисклеймер.** Это публичное описание архитектуры реальной системы, в разработке которой автор участвовал. Конкретные клиенты, доменные имена, финансовые показатели, исходный код и проприетарные детали реализации не раскрываются. Содержание ограничено архитектурными решениями и принципами, обсуждаемыми в публичном поле для систем такого назначения.

Описание ключевых архитектурных решений: контекст (что решали) и реализация (как сделано).

---

## 1 · Обработка звонка — одна Celery-задача с оркестратором

### Контекст

Обработка одного звонка — это последовательность из 5–7 шагов: ingestion, preprocessing, STT, диаризация, LLM-анализ, postprocessing, push результата.

### Реализация

Обработка одного звонка — это один Celery-таск, внутри которого `PipelineOrchestrator` последовательно вызывает шаги. Итоговое состояние — `call_analysis.status` (`pending` / `processing` / `completed` / `failed`).

Recovery работает через идемпотентность записи: `INSERT ... ON CONFLICT (external_call_id, project_id) DO UPDATE`. Любой повторный запуск обработки того же звонка обновляет существующую запись, не создаёт дубль. Перезапустить упавший звонок можно безопасно: вручную из админки или через auto-recovery-таск, который детектит «застрявшие» в `processing` записи и возвращает их в очередь.

## 2 · AI-пресеты как зашифрованный JSON в `system_config`

### Контекст

Разные клиенты оценивают звонки по разным критериям (продажи, техподдержка, претензии) и подключают разные модели (топовые LLM или дешёвые self-hosted). Нужна сущность «AI-пресет» с возможностью переключаться без релизов.

### Реализация

Пресеты хранятся в таблице `system_config` под фиксированным ключом `ai_presets`. Значение — Fernet-зашифрованный JSON-массив. В UI создаются, редактируются, активируются. Чувствительные поля (`stt_api_key`, `diarization_api_key`, `llm_api_key`) маскируются в ответах API; полный текст видим только в момент создания / редактирования. К проекту привязывается активный пресет по id; можно создавать несколько и переключаться.

## 3 · Staged-таблица для ручного отбора звонков перед обработкой

### Контекст

Не все звонки из CRM полезно обрабатывать. Часть может быть короткими, тестовыми или нерелевантными. STT и LLM стоят денег; в некоторых проектах админ хочет сам решать, какие звонки идут в обработку.

### Реализация

При получении звонка из CRM — INSERT в `staged_calls` со статусом `pending`. Если у проекта `auto_process=True`, звонок сразу enqueue'ится в Celery; иначе — ждёт ручного действия админа. Status'ы: `pending | processing | completed | failed | skipped`. Поведение управляется одним флагом на проекте (полностью авто vs ручной отбор).

## 4 · Project.config_json (JSONB) для настроек проекта

### Контекст

Проект хранит несколько кластеров настроек: подключение к CRM, выбор AI-пресета и параметры промта, контроль доступа, нотификации в Telegram. Каждое из этих может развиваться независимо.

### Реализация

`Project.config_json` (JSONB) хранит вложенную структуру: `connection`, `ai`, `prompt`, `access_control`, `telegram`. Pydantic-модели (`CRMConnectionConfig`, `AIConfig`, `PromptConfig`, `AccessControlConfig`, `TelegramConfig`) валидируют структуру при сохранении и чтении. Изменения в конфиге журналируются в audit log.

## 5 · Готовность к on-premise через провайдер-абстракции

### Контекст

Часть потенциальных клиентов работает в compliance-чувствительных отраслях (здравоохранение, финансы, государственный сектор), где аудио и транскрипты не могут покидать периметр клиента. Это означает, что внешние SaaS-провайдеры STT/LLM использовать нельзя — нужны локальные модели в инфраструктуре клиента.

### Реализация

Архитектура использует единую кодовую базу для SaaS и on-premise. Различия — на уровне конфига:

- STT/LLM-провайдеры — через абстракцию в AI-пресете, переключаются на локальные сервисы без изменения кода.
- Fernet-шифрование секретов — локальное, не требует внешних KMS.
- БД, очереди, файловое хранилище — всё локализуемо.
- CRM-интеграции — на стороне клиента (если есть CRM в его периметре).

Production-инсталляция сейчас в SaaS-режиме. Для запуска on-premise у клиента — упаковка деплой-пакета (Docker-образы или systemd-units), документация по установке, compliance-проверки под отрасль.

## 6 · Webhook + polling в одном CRM-интерфейсе

### Контекст

Разные CRM по-разному отдают новые звонки: одни шлют webhook на наш endpoint при появлении звонка, другие требуют опроса (polling) по расписанию.

### Реализация

Реестр (`integrations/registry.py`) хранит реализации под каждый CRM-тип. Каждая реализация знает: умеет ли она webhook, умеет ли polling, какой формат данных. После приёма (любым способом) звонок попадает в `staged_calls` — дальше единый путь.

Тип CRM (webhook vs polling) определяется в адаптере; manual sync для webhook-типа недоступен (валидируется в API).

## 7 · Fernet-шифрование чувствительных полей в БД

### Контекст

В системе несколько классов чувствительных данных: API-ключи и токены STT/LLM-провайдеров (в AI-пресетах), credentials CRM-интеграций, токены/пароли прокси.

### Реализация

Чувствительные значения шифруются вызовом `encrypt_value()` перед записью и расшифровываются на чтении. Encryption key хранится в env; ключи провайдеров и `system_config.value` для AI-пресетов — целиком зашифрованные JSON-объекты. В API-ответах ключи маскируются (`****abcd`); открытый текст видим только в момент создания/редактирования. Бэкапы и физические дампы БД не содержат plaintext.
